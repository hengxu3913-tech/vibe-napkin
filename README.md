## 🎬 一个"动嘴型 PM"的血泪进化史

> 这是一个关于「阿强」的故事。  
> 如果你也是那种对着 AI 指点江山、用嘴把功能"说"出来的人——你会懂。

### 🎮 第一章：蜜月期——游戏存档般的丝滑

阿强不懂代码，但他懂业务。  
他打开 Cursor，动动嘴："给我爬一下这个商品的字段。" AI 动了。  
"把爬到的字段做个表单。" AI 又动了。

每趟通一个功能，阿强就兴奋地往 `plan.md` 里记一笔。  
像打 RPG 一样：到哪关存档，下次读档继续。  
项目推进，丝滑无比。

### 🐻 第二章：窒息期——黑瞎子掰玉米

项目大了。MySQL、ECS、Redis、Playwright 全堆上去了。  
`plan.md` 从 60kb 涨到 100kb。

AI 开始不对劲了——  
读完文档再读代码，1M 上下文剩 200k。  
按下葫芦起来瓢。前脚修好后脚忘。  
**文档越大，AI 越傻。**

阿强就像黑瞎子掰玉米：抓一棒，掉一棒。

### ⏳ 第三章：绝望期——知识库的"时间小偷"

"上知识库！"阿强想。  
他搭了 LightTag，准备把存量文档做成知识图谱。  
结果：**建索引要 8-10 小时。**

需求一天改三回。索引速度赶不上改需求的速度。  
知识库直接废了。

### 💡 第四章：顿悟期——纵向业务单元

某天，大模型跟阿强说了一句话：  
> "你的管理方式是面向人的矩阵式文档，不是面向 AI 的。"

阿强把矩阵文档全砍了。  
每个功能点一个 ≤200 行的 `.md`，纵向贯穿——  
产品、技术、部署、数据库，一条闭环写完。  
公共部分切成"基线约束"。  
配合 zvec 秒级增量索引 + MCP 轻量查询。

**AI 再也没失忆过。**

## 🧻 核心理念：三轨并行 + 擦嘴哲学

### 三轨并行（不是先后顺序，是同时）

```
基线约束  ──── 游戏物理引擎（Git怎么用、密钥在哪、技术栈）
业务单元  ──── 关卡产物（纵向贯穿，≤200行）
进度存档  ──── 存档点（随时读档继续）
```

### 擦嘴两步走

开发就像吃饭。你动嘴（Vibe）大快朵颐，AI 给你端菜上桌。  
吃饱了，满嘴碎渣——这时候你需要擦嘴。

| 动作 | 命令 | 耗时 | 阻塞 MCP？ | 适合场景 |
|------|------|:----:|:---------:|---------|
| 🧻 **擦嘴** | `vibe-napkin wipe` | <100ms | ❌ 否 | **猛猛推功能**：验收完就擦，不打断情绪 |
| 🗑️ **归档** | `vibe-napkin tidy` | 几秒~几分钟 | ✅ 是（安全锁） | **稳扎稳打**：中场休息/下班前跑，知识库彻底沉淀 |

**PM 主导节奏，你说了算：**

```
🔁 猛猛推功能模式（快速迭代）
  开发 → 验收 → wipe → 下一功能
  （只同步 .md 文档，毫秒级，不卡顿）

🔂 稳扎稳打模式（完整沉淀）
  开发 → 验收 → wipe → tidy → save → 下一关
  （同步文档 + 归档知识库 + 存档）
```

想冲进度就只 wipe 不 tidy，一轮接一轮猛推。  
想稳一稳就 wipe + tidy + save，让知识库跟上节奏。  
**PM 掌控节奏，工具不替你做决定。**

## 🚀 快速上手

```bash
pip install vibe-napkin
```

```bash
# 1. 支桌子
cd my-project
git init
vibe-napkin init

# 2. 猛猛推功能
开发功能 → 验收 → vibe-napkin wipe → 下一功能

# 3. 稳扎稳打（中场休息/下班前）
开发功能 → 验收 → vibe-napkin wipe
vibe-napkin tidy                     # 归档知识库
vibe-napkin save -l "商品采集完成"    # 存档

# 4. 下次继续
vibe-napkin load 2026-06-29-001-商品采集完成
```

## 🎮 六大命令

| 命令 | 动作 | 说明 |
|:-----|:-----|:------|
| `vibe-napkin init` | ⛺ 支桌子 | 初始化环境 + 模板底座 + 部署 MCP 服务 |
| `vibe-napkin wipe` | 🧻 擦嘴 | 扫描变更 → 自动校验格式 → 同步业务单元文档 |
| `vibe-napkin tidy` | 🗑️ 归档 | 锁检测 → 用户确认 → 增量更新向量知识库 |
| `vibe-napkin save` | 💾 存档 | 保存进度快照（日期+序号+标签） |
| `vibe-napkin load` | 📂 读档 | 恢复到指定存档点，输出上下文 |
| `vibe-napkin list` | 📋 清单 | 查看所有存档 |

## 🧠 业务单元模板

```markdown
# [业务名称]

## 关键词           ← 向量召回用
英文,中文 逗号分隔

## 业务规则          ← 核心逻辑
- 规则 1（一句话说明）

## 代码位置          ← 路径::符号 行号
- 函数/类名：`src/crawler.py::crawl_product` 42

## 关联业务单元       ← 交叉引用
- [数据映射](./数据映射.md)

## 历史决策（近3次）  ← 决策记录
- 2026-06-01（abc123）：决定用 Playwright 而非 API

## 变更触发器        ← 什么情况回来改
- OZON 页面 DOM 结构变更
```

## ⚠️ 防呆机制（实战踩过的坑）

### 🧟 死锁问题

`tidy` 同步知识库时，如果 MCP 正在被 AI 引用，会触发文件锁冲突。  
AI 遇到报错会自作聪明——改 zvec 代码、杀进程、加进程。  
**阿强曾搞出 30+ 个僵尸 zvec 进程。**

**解决方案：**
```
tidy 流程内置"锁协议"：

① 检测 MCP 是否被 AI 占用
② 占用 → 提示用户关闭 AI 侧 MCP 连接
③ 用户确认后 → 执行向量增量更新
④ 自检索引是否正常
⑤ 通知用户重新连接 MCP → 继续编码
```

### 🔒 开发者负责验收

`wipe` 信任你的判断，不会替你验证业务逻辑。  
**功能开发完，充分验收后再 wipe。**
